# SM8845 HTP 通路诊断报告

## 一句话结论

**QAIRT 2.48.40 SDK 的 `QairtSystemBuilder` (Easy API) 在 SM8845 平台上有 bug**：往 `QnnDevice_create` 传了个 8845 拒收的默认配置，导致 build 返回 `QNN_DEVICE_ERROR_INVALID_CONFIG`（我们看到的 30104 = `QAIRT_SYSTEM_BUILDER_ERROR_BUILD_FAILED`，真错误被吞掉）。

**QNN 底层 API 通路完全正常** —— 使用 SDK 官方 `qnn-net-run` 工具在同一台设备上、用同一份 DLC 能成功推理。

**从 app 层跑 HTP 也不行** —— 即使我们绕过 QairtSystemBuilder、切换 unsigned PD 配置、把 skel 放到 app 私有目录 `/data/user/0/com.breeze.qnn/files/htp/`，从 app 进程上下文里 `loadRemoteSymbols` 依然返回 err 4000。原因是 FastRPC daemon 读 skel 用的是 daemon 自己的进程凭据，untrusted_app 上下文没法让它换到 unsigned PD 路径。

**结论**：SM8845 上从 untrusted_app 直接跑 HTP 需要 platform-signed app 或 vendor 提供匹配 SDK 版本的签名 skel，纯 SDK 层面无法绕过。

## 最终工作状态

| 后端 | 状态 | 备注 |
|---|---|---|
| **CPU** | ✅ PASS (~10ms 推理) | 通过 QAIRT SystemBuilder + `Qairt.executeGraph`，用 `getModelInputInfoAt` 拿 tensor 真指针（不是 GraphInfo 的不透明句柄，那样会 SIGSEGV，fault=handle+0x28）。E2E test `com.breeze.qnn.BackendE2ETest#cpuBackend` PASS。 |
| **HTP** | ❌ 阻塞（documented） | `loadRemoteSymbols err 4000` — FastRPC 从 app 上下文读不到匹配的 skel。E2E test SOFT FAIL。 |
| **GPU** | ❌ 阻塞（documented） | 8845 vendor 没配 OpenCL ICD (`/vendor/etc/OpenCL/vendors/*.icd` 缺失)，虽然 `libOpenCL_adreno.so` 存在。E2E test SOFT FAIL。 |

## 已解决 & 未解决

**已解决**：
- CPU 后端在 App 里可用（已修的关键 bug：`Qairt_executeGraph` 必须用 `getModelInputInfoAt` 返回的真指针，不是 `GraphInfo.getInputAt` 的不透明句柄——否则 SIGSEGV，fault = handle + 0x28）
- 量化 DLC 生成流程（`snpe-dlc-quant.exe` + `snpe-dlc-graph-prepare.exe --htp_archs v81`）通了
- 8845 HTP 底层能力完全 OK（qnn-net-run 验证）

**未解决**：
- App 里 HTP 通路 —— 需要绕过 `QairtSystemBuilder`
- GPU 后端 —— 缺 OpenCL ICD 注册

## HTP 通路修复方向（refactor 计划）

放弃 `QairtSystemBuilder.build()`，改用 SampleApp（`examples/QAIRT/SampleApp/src/QairtSampleApp.cpp`）的低层 API：

```
qairt::Api → dlopen 后端 .so (libQairtHtp.so / libQairtCpu.so)
  → 从后端 .so 里 dlsym("QairtInterface_getInterface")
  → 拿 QairtBackend_V1_t / QairtContext_V1_t / QairtGraph_V1_t 三张接口表
  → Backend.create(logHandle, nullptr, &backendHandle)
  → Context.create(backendHandle, nullptr, &contextHandle)
  → SystemDlc.composeGraphs(dlcHandle, nullptr, 0, backendHandle, contextHandle, backendInterfaceFn, &graphInfoSet)
  → Context.retrieveGraph(name, &graphHandle) 拿每张图
执行时：
  → QairtTensor.create + setId/setName/setDataType/setDims/setTensorMemory 等 ~15 setter
  → QairtGraph.execute(graphHandle, inputTensors, N, outputTensors, N, nullptr, nullptr)
  → 从 output QairtTensor 的 memory 读回结果
```

工作量估计 500-700 行 C++。

## 关键运行时依赖

**jniLibs/arm64-v8a/**：
- Required: libQairtSystem.so, libQairtCpu.so
- Optional: libQairtHtp.so, libQairtGpu.so
- HTP 用：libQnnHtp.so, libQnnHtpV81Stub.so, libQnnHtpV81CalculatorStub.so, libQnnHtpPrepare.so, libQairtHtpPrepare.so
- CPU/GPU 用：libQnnCpu.so, libQnnGpu.so
- 系统：libQnnSystem.so, libQnnModelDlc.so

**HTP Skel 部署**（DSP 侧，通过 FastRPC 加载）：
- 因为 vendor 签名 skel 与 SDK 2.48 不兼容，走 unsigned skel：`libQnnHtpV81Skel.so`（12.6MB, `hexagon-v81/unsigned/`）
- 部署路径：目前放 `/data/local/tmp/qnn_test/lib/`；产品化需打进 assets 由 Kotlin 释放到 `context.filesDir/htp/`
- 环境变量：`ADSP_LIBRARY_PATH=<skel dir>;/vendor/lib/rfsa/adsp;...`（skel dir 优先）
- 关键选项：`unsignedPD:ON`

## 量化 DLC 生成命令

```powershell
# 量化 FP32 DLC → INT8
D:\workspace\docs\qti\qairt\2.48.40.260702\bin\x86_64-windows-msvc\snpe-dlc-quant.exe `
  --input_dlc  model.dlc `
  --input_list input_list.txt `
  --output_dlc model_int8.dlc

# 校准数据：input_list.txt 每行一个 raw 文件绝对路径，raw 文件是 float32 二进制，
# 大小 = 输入 tensor bytes（例：1×3×224×224 float32 = 602112 字节）
# 5-10 张随机数据即可（demo 只验证链路，不追求精度）
```

**注意**：offline HTP cache（`snpe-dlc-graph-prepare --htp_archs v81`）在这台设备上**加载不了**（`Could not create context from binary`）。用**不带 cache 的 INT8 DLC** 让 QNN JIT 编译即可，性能损失微小（首次加载 +100~200ms）。
